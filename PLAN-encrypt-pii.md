# План: шифрование ПД (телефоны, медданные) — GDPR/ЗУ про ЗПД

**Статус:** на согласование · **Риск:** ВЫСОКИЙ (ломает поиск по телефону, дубли, все phone-зависимости)
**Правило Босса:** не сломать рабочее. Только с полным бэкапом БД + поэтапно + откат.

## Зачем
Аудит v2 (юрист): телефоны клиентов и медданные (аллергии) хранятся открытым текстом.
Для продажи как SaaS в ЕС — шифрование ПД at-rest желательно (не строго обязательно при RLS+TLS, но усиливает).

## Почему НЕ наспех
- Поиск/дедуп клиентов идёт ПО ТЕЛЕФОНУ (`WHERE phone=`, ON CONFLICT (tenant_id, phone)). Зашифрованный телефон нельзя искать напрямую.
- Телефон используют десятки мест: booking, mobile, crm-card, mono-webhook, reminders, экспорт.
- Ошибка = потеря доступа к клиентам или дубли. Это хуже, чем текущее состояние (данные и так изолированы RLS + TLS в транзите + БД Neon шифрует диск).

## Подход (envelope encryption + blind index)
1. **Миграция**: добавить `phone_enc bytea` (шифротекст), `phone_bidx text` (HMAC-хеш для поиска), оставить `phone` временно.
2. **Крипто-слой** (`lib/pii-crypto.js`): AES-256-GCM с ключом из env (`PII_KEY`), blind index = HMAC-SHA256(phone, PII_KEY) для точного поиска/дедупа.
3. **Двойная запись**: писать и `phone` (старое), и `phone_enc`+`phone_bidx` — переходный период, ничего не ломается.
4. **Бэкфилл**: скрипт шифрует существующие телефоны батчами (с бэкапом до).
5. **Переключить чтение/поиск** на `phone_bidx` (WHERE phone_bidx=hmac($1)); вывод — расшифровка `phone_enc`.
6. **Убрать** открытый `phone` только после полной верификации (недели наблюдения).
7. Медданные (medical_cards) — то же, но без blind index (не ищутся) — проще.

## Ключ
- `PII_KEY` в env Render (32 байта). Ротация — отдельная процедура (перешифровка).
- Потеря ключа = потеря доступа к ПД → ключ в бэкап-хранилище Босса.

## Этапы (каждый — отдельный деплой + проверка)
1. Миграция колонок + lib/pii-crypto (не используется) — безопасно.
2. Двойная запись в booking/mobile/signup.
3. Бэкфилл-скрипт (после git-бэкапа БД).
4. Переключение поиска на blind index — самый рискованный, тщательный тест дедупа.
5. Удаление plaintext phone.

## Критерий
Поиск клиента по телефону, дедуп (ON CONFLICT), онлайн-запись, mono-webhook, экспорт — работают идентично. Ноль потерянных/задвоенных клиентов.

## Оценка: 1-2 недели фокусной работы. Делать ПОСЛЕ первых продаж, не блокер запуска.
Текущая защита ПД (RLS-изоляция + TLS + шифрование диска Neon + erasure + consent) достаточна для старта.
