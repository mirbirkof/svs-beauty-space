# Переезд CRM на другой сервер/базу — чтобы НИЧЕГО не потерялось

Три места, где живут данные. Потерять можно любое — переносить надо все три.

| Что | Где живёт | Как переносится |
|---|---|---|
| **Данные** (клиенты, записи, склад, касса) | Postgres (Neon) | бэкап БД / смена строки подключения |
| **Файлы** (фото, документы, AI-видео) | диск сервера (`uploads/`) | отдельный архив — в бэкап БД НЕ входят |
| **Ключи шифрования** | переменные окружения | скопировать 1:1, иначе данные не расшифровать |

---

## ⚠ Самое опасное — ключи шифрования

В БД зашифрованы телефоны клиентов (`PII_KEY`), токены интеграций
(`INTEGRATION_ENC_KEY`), лицензии (`LICENSE_KEY_SECRET`).

Если на новом сервере сгенерировать **новые** ключи вместо переноса старых —
данные останутся в базе, но расшифровать их будет **невозможно навсегда**.
Телефоны клиентов пропадут, поиск по ним сломается.

**Правило: эти переменные копируются со старого сервера как есть, буква в букву.**
Полный список с пометкой `[SECRET-CARRY]` — в `backend/.env.example`.

## Порядок переезда

### На СТАРОМ сервере (снять всё)
1. **Эталон состояния** (сколько чего было — чтобы потом сверить):
   ```
   cd backend && node -r dotenv/config scripts/verify-migration.js --save /tmp/baseline.json
   ```
2. **Бэкап БД** (снимок всех таблиц):
   ```
   node -r dotenv/config scripts/db-backup.js
   ```
   Либо, если БД остаётся на том же Neon — переносить не нужно, только строку `DATABASE_URL`.
3. **Файлы**:
   ```
   bash scripts/uploads-pack.sh pack
   ```
   Получится `uploads-ДАТА.tar.gz` — забрать на новый сервер.
4. **Переменные окружения** — выписать все значения из Render Dashboard
   (Environment). Особенно `[SECRET-CARRY]` из `.env.example`.

### На НОВОМ сервере (развернуть)
1. Подключить репозиторий как **Render Blueprint** (`render.yaml`) — создаст
   сервис + постоянный диск `/data` для файлов автоматически.
2. Вписать все переменные окружения (`sync:false` из блупринта задаются руками).
   Ключи шифрования — **1:1 со старого**.
3. База: указать `DATABASE_URL`. Если новая БД — восстановить из бэкапа; если
   тот же Neon — просто та же строка. Схема накатится сама (миграции при старте).
4. Файлы:
   ```
   UPLOADS_DIR=/data/uploads bash scripts/uploads-pack.sh unpack uploads-ДАТА.tar.gz
   ```

### Проверка — «ничего не потерялось?»
На новом сервере:
```
node -r dotenv/config scripts/verify-migration.js --compare /tmp/baseline.json
```
Скрипт сверит число миграций и счётчики таблиц со старым сервером, проверит что
`PII_KEY` реально расшифровывает телефоны и что файлы на месте.
**Код 0 и «Всё на месте» = переезд чистый. Код 1 = показывает что именно пропало.**

---

## Почему теперь безопаснее, чем было
- Файлы больше не привязаны к коду: путь задаётся `UPLOADS_DIR` → постоянный
  диск Render (`/data`), переживает редеплой и переезд.
- `render.yaml` разворачивает инфраструктуру одинаково — не собирать руками.
- `.env.example` содержит все 90+ переменных с пометкой что терять нельзя.
- `verify-migration.js` даёт доказательство целостности, а не «вроде работает».
