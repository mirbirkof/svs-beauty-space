# Живой организм: основной + резервный сервер с автопереключением

Цель: когда на основном Render кончаются лимиты (или он падает) — трафик сам
уходит на резервный, данные те же, ничего не теряется. Резервный до этого
момента спит и лимиты не жжёт.

## Из чего состоит организм (3 слоя)

```
                   ┌─────────────────────────┐
   клиенты  ───►   │  ДИСПЕТЧЕР (health-LB)   │   следит за /health основного
                   └───────────┬─────────────┘   упал → шлёт на резервный
                       ┌───────┴────────┐
                       ▼                ▼
              ┌────────────────┐  ┌────────────────┐
              │ Render ОСНОВНОЙ│  │ Render РЕЗЕРВ   │  спит (scale-to-zero),
              │  (активен)     │  │ (в ожидании)   │  просыпается по запросу
              └───────┬────────┘  └────────┬───────┘
                      └────────┬───────────┘
                     ОБЩИЕ данные (один источник):
              ┌──────────────┐   ┌────────────────────┐
              │ Neon Postgres│   │ S3-облако (файлы)   │
              │ (одна база)  │   │ фото/док/видео      │
              └──────────────┘   └────────────────────┘
```

## Что УЖЕ готово (сделано в коде)

[+] **База — общая.** Оба сервера смотрят в один Neon. Данные синхронны сами
    собой, писать/читать может любой.
[+] **Файлы — общие.** Загруженный файл дублируется в S3-облако; любой сервер,
    у которого его нет локально, тянет из облака. Диск = кэш. (commit 9ab48dd)
[+] **Одинаковое развёртывание.** `render.yaml` разворачивает сервис идентично
    на любом Render-аккаунте — резерв поднимается тем же блупринтом.
[+] **Health-эндпоинт.** `/health` отдаёт `{ok, rev}` — диспетчеру есть за чем
    следить.
[+] **Ключи шифрования переносимы 1:1** — на обоих серверах данные читаются.

Итог: **слой данных уже «один организм»** — что основной, что резервный видят
одно и то же. Осталось поставить диспетчер и поднять второй Render.

## Что нужно подключить (требует твоих аккаунтов — 3 шага)

### Шаг 1. S3-облако для файлов (активирует общее хранилище)
Завести бакет и ключи, вписать в оба Render как `BACKUP_S3_*` (см. `.env.example`).
Рекомендую **Cloudflare R2** (бесплатно до 10 ГБ, без платы за трафик) или
**Backblaze B2**. Код уже готов — заработает сразу как впишешь ключи.

### Шаг 2. Резервный Render (в режиме сна)
- Создать 2-й Web Service из этого же репозитория (можно на другом Render-
  аккаунте — тогда лимиты часов считаются отдельно).
- Те же переменные окружения, что у основного (включая ключи шифрования и S3).
- Free-tier сам засыпает после 15 мин простоя → лимиты не расходуются, пока
  трафик на основном.

### Шаг 3. Диспетчер (health-фейловер)
Ставит перед серверами и переключает трафик. Варианты:
- **Cloudflare Load Balancer** (~$5/мес): health-check на `/health` основного,
  при падении — на резервный. Публичный (не туннель) → работает у всех клиентов
  независимо от VPN. Нужен домен на Cloudflare DNS.
- **DNS-failover** (dnsmadeeasy/ClouDNS): проще, но переключение медленнее
  (минуты, из-за DNS-кэша).

## Про лимиты и «спящий резерв»
- Резерв на free-tier спит, пока не придёт запрос → часы не тратятся.
- Диспетчер шлёт на резерв только когда основной не отвечает на `/health`.
- Как только основной поднялся (новый месяц/лимиты) — диспетчер вернёт трафик.
- Хочешь совсем без простоя переключения — резерв держать «тёплым» (пинг раз в
  10 мин), но тогда он тратит часы. Компромисс на твой выбор.

## Чего это НЕ делает (честно)
- Не реплицирует саму базу Neon (она и так одна общая — реплика нужна только
  если хочешь пережить падение самого Neon; это отдельный уровень).
- Автопереключение появляется только после Шага 3 (диспетчер). До него —
  переезд ручной по MIGRATION.md, но данные уже общие.
